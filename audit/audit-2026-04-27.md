# Rex Backend Audit - 2026-04-27

## Full Audit Report

I audited every file in `backend/app/`.

## 1. What's Good

- The backend is small and understandable. For a starter project, the folder layout is fine: `routes/`, `services/`, `models/`, `database.py`.
- `routes/chat.py` has the correct high-level flow: validate input, create/resume conversation, save user message, build context, call AI, save assistant response, return result.
- `ai_service.py` is pointed at local Ollama and avoids API keys.
- `file_service.py` is simple and supports only `.txt`, `.md`, `.csv`, which is the right scope for now.
- SQLite is acceptable for the current stage.
- The file content is now correctly passed as a user-context message, not injected into the system prompt.

## 2. What's Wrong / Smelly

- `database.py` is stale and actively conflicts with `memory_service.py`. It still creates the old `messages` table with no `conversation_id`, while `memory_service.py` owns the newer schema. This is the messiest part.
- `main.py` calls `init_database()`, which creates the old schema before the memory system has a chance to create the right one.
- `memory_service.py` calls `ensure_memory_tables()` inside almost every function. That is noisy and inefficient. Schema setup should happen once at startup.
- `routes/chat.py` is doing too much: request parsing, validation, conversation selection, file context assembly, memory writes, AI call, response shaping.
- `ChatRequest.file` in `models/chat.py` is just a string filename, not an uploaded file. It is slightly misleading.
- The API switched to multipart form input for `/chat`, which means a normal JSON `POST /chat` no longer works unless another endpoint or compatibility path is added.
- Services return raw `dict` objects everywhere. That works, but it is loose and easy to break later.
- `FileService` raises `HTTPException`, which couples a service layer directly to FastAPI. It is acceptable for a tiny app, but not clean if this grows.
- `AIService` hardcodes the Ollama URL, model, and timeout. Fine for now, but it will become annoying quickly.
- Error handling in `AIService` returns an error string as Rex's response. That means failures get saved as assistant messages, which is bad memory hygiene.

## 3. Major Issues

- **Broken schema ownership:** `database.py` and `memory_service.py` both define database setup, but with different schemas. This can create weird migration behavior and empty placeholder conversations.
- **Blocking AI call inside async route:** `routes/chat.py` is async, but `AIService` uses blocking `urlopen()`. While Rex waits for Ollama, the event loop is blocked.
- **Partial write risk:** `/chat` saves the user message before the AI call. If Ollama fails, the user message remains saved and the error text may be saved as Rex's reply.
- **No file size limit:** A huge `.txt`, `.md`, or `.csv` can be read fully into memory and dumped into the AI context.
- **No context budget:** Last 20 messages plus full file content can easily exceed model context or make responses slow/expensive locally.
- **No real tests:** Smoke tests were run manually, but there are no durable backend tests in the repo.
- **Deprecated startup style:** `@app.on_event("startup")` and `@router.on_event("startup")` are old FastAPI patterns. It works, but it should eventually move to lifespan startup.

## 4. Suggested Improvements

- Make `memory_service.py` the only owner of the database schema, or move all schema setup into `database.py`. Do not keep both.
- Remove the old `init_database()` behavior from `database.py` or update it to create the real `conversations` and `messages` schema.
- Keep `/chat` thin. Add a small orchestration function later, like `chat_service.py`, once the flow grows.
- Keep file upload context separate from persisted memory for now, but structure it clearly as "temporary context".
- Add a file size cap, probably something tiny like 1 MB for now.
- Change AI failures so they return a proper HTTP error or controlled response without saving bad assistant memory.
- Use typed message models instead of loose `list[dict]`.
- Add backend tests for:
  - new conversation creation
  - continuing an existing conversation
  - last-20 message retrieval
  - file upload accepted/rejected
  - Ollama failure behavior

## Cleanup Priority

1. Fix database schema ownership first. `database.py` and `memory_service.py` conflict. This is the top cleanup.
2. Move schema setup to startup only. Stop calling `ensure_memory_tables()` inside every memory function.
3. Fix AI error handling. Do not save "Rex could not reach local Ollama..." as memory.
4. Add file size and context limits. Prevent huge uploads from polluting context.
5. Clean `routes/chat.py`. Keep the endpoint readable and move helper flow into a small service only if it starts growing.
6. Add real tests. The backend is now complex enough that manual smoke tests are not enough.
7. Make Ollama config adjustable. Model, URL, and timeout should not stay hardcoded forever.
